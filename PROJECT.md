# Impression OS — Project Charter

> The definitive statement of what Impression OS is, why it exists, how it is
> structured, and — just as importantly — what it will never be.

---

## 1. Vision

Impression OS is an **AI-native framework for generating professional WordPress
websites using Elementor Pro**.

The ambition is not to produce "an AI website builder." Countless tools already
turn a prompt into a page. The ambition is to encode the judgement of a senior
design and front-end team into a system that an AI agent can execute
faithfully — so that the output is not *plausible*, but *professional*: something
a studio would ship to a paying client without embarrassment.

We treat this as an operating-system problem, not a template problem. An OS
provides primitives, contracts, and a runtime; applications are composed on top.
Impression OS provides design primitives (tokens, components), design contracts
(foundation, accessibility), and a runtime (the builder), so that individual
websites become *applications* of the system rather than bespoke one-offs.

## 2. Why this exists

Three gaps make today's AI web generation unfit for professional work:

1. **Non-determinism.** Models improvise design values inline. Two runs of the
   same brief yield different spacing, type, and color. Quality is luck.
2. **Incoherence at scale.** A single section can look good; a twelve-page site
   drifts, because there is no shared source of truth binding the pages together.
3. **Inaccessible output.** Contrast failures, broken heading order, and missing
   focus states are the norm, because accessibility is treated as a post-hoc
   review rather than a system constraint.

Impression OS closes all three by moving every design decision *out* of the
generation step and *into* a versioned, testable system that the generator merely
composes.

## 3. What it is

- A **design system expressed as data** — tokens and components that are
  machine-legible and human-authored.
- A **generation engine** — the builder — that compiles that system into an
  Elementor Pro kit (global styles + templates) for WordPress.
- A **prompt architecture** — a disciplined interface that turns a client brief
  into a deterministic build plan against the system.
- A **quality harness** — schema validation, accessibility contracts, and visual
  regression that make "professional" a testable property, not an opinion.

## 4. What it is NOT

- ❌ **Not a website.** No single site is the deliverable; the system is.
- ❌ **Not an Elementor template.** Templates are an *output* of the OS, not the OS.
- ❌ **Not a theme or a page builder plugin.** It targets Elementor Pro; it does
  not replace or fork it.
- ❌ **Not a black box.** The generated kit is clean and fully editable in
  Elementor. There is no proprietary runtime the client depends on.
- ❌ **Not a prompt-only trick.** The intelligence lives in the system layers, not
  in a single clever prompt.

## 5. Architecture

Impression OS is organised as a strict dependency stack. A layer may depend only
on the layers below it, never on the layers above.

```
┌───────────────────────────────────────────────────────────────┐
│  prompts/     Intent → build plan. The AI interface.           │
├───────────────────────────────────────────────────────────────┤
│  recipes/     Composed sections built from components.          │
├───────────────────────────────────────────────────────────────┤
│  components/  Atomic units mapped to Elementor widgets.         │
├───────────────────────────────────────────────────────────────┤
│  foundation/  Design laws: grid, rhythm, hierarchy, a11y.       │
├───────────────────────────────────────────────────────────────┤
│  tokens/      Design decisions as data. The source of truth.    │
└───────────────────────────────────────────────────────────────┘
        ▲
        │  compiled by
        │
   builder/  ──▶  Elementor Pro Kit  (globals + section/page templates)
```

Supporting subsystems:

- **`tools/`** — authoring, validation, and shipping utilities (CLI).
- **`examples/`** — full sites generated by the OS, committed as living proof.
- **`tests/`** — the executable definition of "correct."
- **`docs/`** — the reasoning behind every decision.

### The generation pipeline

1. **Brief** — a structured description of the client, goals, and content.
2. **Plan** — `prompts/` converts the brief into a build plan: which recipes,
   which content, which token theme.
3. **Compile** — `builder/` resolves the plan against tokens, foundation,
   components, and recipes.
4. **Emit** — the builder outputs an Elementor Pro kit: global colors/fonts plus
   importable templates.
5. **Verify** — `tests/` validate schema, contrast, semantics, and visual diffs.
6. **Ship** — the kit is imported into WordPress; the site is fully editable.

## 6. Design principles

| # | Principle | Consequence |
| - | --------- | ----------- |
| 1 | Design decisions are data. | No hard-coded values above the token layer. |
| 2 | Compose, don't improvise. | The generator selects primitives; it never invents design values. |
| 3 | Deterministic by default. | Same brief + same version → same site. |
| 4 | Accessibility is a contract. | Contrast, semantics, and focus are enforced, not reviewed. |
| 5 | The kit is the artifact. | Output is clean, editable Elementor — no lock-in. |
| 6 | One source of truth per decision. | A value is owned by exactly one layer. |
| 7 | Documentation is part of the build. | An undocumented primitive is an incomplete primitive. |

## 7. Scope

### In scope
- Marketing and business websites (single- and multi-page) built on WordPress + Elementor Pro.
- A token-driven, themeable design system with light/dark and brand theming.
- Deterministic generation from a structured brief.
- Accessibility to WCAG 2.1 AA as a system guarantee.

### Out of scope (for now)
- E-commerce / WooCommerce flows.
- Application-style dynamic web apps.
- Non-Elementor or non-WordPress targets.
- Content strategy and copywriting as a product (the system consumes content; it does not replace a copywriter).

Out-of-scope items are candidates for future phases, not permanent exclusions.

## 8. Audience

- **AI agents** operating the system to build sites (see [CLAUDE.md](CLAUDE.md)).
- **Design engineers** authoring and extending tokens, components, and recipes.
- **Studios and operators** who ship the resulting kits to clients.

## 9. Definition of done (for the system)

Impression OS reaches its first meaningful milestone when a structured brief can
be turned into an importable Elementor Pro kit that:

1. passes every schema, accessibility, and visual-regression test;
2. renders a coherent multi-section page with a real type and spacing scale;
3. is fully editable in Elementor with no orphaned or broken styles; and
4. reproduces byte-stable given the same brief and version.

See [ROADMAP.md](ROADMAP.md) for the path there.

## 10. Reference points

We hold ourselves to the standard set by systems that made good decisions
*inevitable* rather than *optional*:

- **Tailwind CSS** — constraints as a design system; utility as composition.
- **shadcn/ui** — components you own and can read, not a dependency you rent.
- **Laravel** — developer experience and convention as first-class product concerns.

If a decision in this repository would embarrass those teams, it is the wrong
decision.
