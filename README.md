# Impression OS

**An AI-native operating system for generating professional WordPress websites with Elementor Pro.**

[![Status](https://img.shields.io/badge/status-alpha-yellow)](ROADMAP.md)
[![Version](https://img.shields.io/badge/version-0.1.0-blue)](ROADMAP.md)
[![License](https://img.shields.io/badge/license-MIT-green)](LICENSE)

---

Impression OS is not a website. It is not an Elementor template, a theme, or a
starter kit. It is the **system that produces** those things — a structured,
machine-legible design language paired with a generation engine, so that an AI
agent can assemble production-grade WordPress sites the way a senior design team
would: from tokens, to components, to composed layouts, to a shippable Elementor
Pro kit.

Where a template answers *one* brief, Impression OS answers *any* brief within
its design system — consistently, accessibly, and without regressing quality as
scope grows.

## The problem

AI can already generate WordPress pages. What it cannot reliably do is generate
pages that a professional studio would sign off on: coherent spacing, a real
typographic scale, accessible contrast, semantic structure, and design decisions
that hold together across an entire site rather than a single hero section.

The failure mode is always the same — the model improvises design values inline,
so quality is non-deterministic and drifts from page to page. Impression OS
removes the improvisation. Every design decision lives as data. The model
*composes* known-good primitives instead of *inventing* new ones.

## The model

Impression OS is layered. Each layer is authored once, versioned, and consumed
by the layer above it. Nothing higher up is allowed to hard-code a value that a
lower layer already owns.

```
  Prompts        The intent layer — how a brief becomes a build plan
     │
  Recipes        Composed sections (hero, pricing, testimonial grid, footer)
     │
  Components      Atomic units (button, card, heading, media object)
     │
  Foundation     The design laws (grid, rhythm, hierarchy, a11y contracts)
     │
  Tokens         Design decisions as data (color, type, space, radius, shadow)
```

The **Builder** is the compiler that walks these layers and emits an **Elementor
Pro kit** — global styles plus page/section templates — ready to import into
WordPress. **Tools**, **Examples**, and **Tests** support and prove the pipeline.

```
  brief ─▶ Prompts ─▶ Builder ─▶ [ Tokens + Foundation + Components + Recipes ]
                          │
                          ▼
              Elementor Pro Kit  (globals + templates)  ─▶  WordPress site
```

## Principles

1. **Design decisions are data.** If a value can be reused, it is a token — never a literal in a component or recipe.
2. **Compose, don't improvise.** Generation selects from known-good primitives; it does not invent new design values at build time.
3. **Deterministic by default.** The same brief and the same version produce the same site. Quality does not depend on luck.
4. **Accessibility is a contract, not a review step.** Contrast, semantics, and focus order are enforced by the system, not left to the author.
5. **The kit is the artifact.** The output is a clean, editable Elementor Pro kit — no proprietary lock-in, no unmaintainable markup.

## Status

Impression OS is in **alpha**. All eight foundational phases are implemented: the
token system, the design-law foundation, components, recipes, the intent/prompt
layer, the builder (a zero-dependency compiler), a CI-gated quality harness, and a
first site generated end-to-end.

A single brief compiles to a complete, byte-stable Elementor Pro kit — see the
[Northwind example](examples/northwind/), reproduced and verified on every change.
The one remaining manual step is importing a kit into a live WordPress + Elementor
environment (documented, not yet automated). See the [Roadmap](ROADMAP.md) for
what's next and [PROJECT.md](PROJECT.md) for the full vision.

**Quick start** (via the [`impression` CLI](tools/)):

```bash
node tools/bin/impression.js studio                             # the GUI: brief → live preview → kit
node tools/bin/impression.js list recipes                       # what the system offers
node tools/bin/impression.js validate                           # check every data artifact
node tools/bin/impression.js build examples/northwind/plan.json --out dist    # generate a kit
node tools/bin/impression.js preview examples/northwind/plan.json --out preview  # see it as HTML
node --test                                                     # run the quality harness
```

The **[Studio](tools/)** is a local, zero-dependency web app: fill a brief on the
left, watch the site render live on the right (including a contrast-safe custom
brand color), and download the Elementor kit as a `.zip`.
